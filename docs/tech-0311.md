# Verge Browser 技术文档

## 1. 文档目的

本文档定义 Verge Browser 后续实现的统一技术基线。

约束原则：

- 用户可自部署，部署时配置单一 admin token，所有业务 API 默认要求鉴权，未携带 token 返回 401。鉴权只做最小实现，不追求产品级 IAM。
- 保持现有 `/sandboxes/{sandbox_id}/...` 路由模型。
- 把 API Server 与 Runtime 视为一个整体系统设计。
- 优先做小而硬的能力，不引入重产品化设计。

## 2. 产品目标

项目目标不是做通用浏览器云平台，而是做一个可自部署、agent-first、同时支持人类接管的浏览器 sandbox 系统。

核心目标：

1. 用户可自部署，并在部署时配置单一 admin token。
2. 所有业务 API 默认都要求鉴权；未携带 token 时返回 `401 Unauthorized`。
3. 提供一个管理页面（Vite + React），让用户能查看和管理全部 sandbox。
4. 提供面向 agent 的 SDK 与 CLI，名称固定为 `verge-browser`，默认从环境变量 `VERGE_BROWSER_TOKEN` 读取 token。
5. 支持 VNC 人工接管与 Playwright/CDP 自动化协同工作。
6. 每个 sandbox 支持 `alias`，便于人和 agent 用稳定名称访问。

明确不做：

- 多用户账号体系
- 多角色权限模型
- JWT 登录流
- 匿名业务访问

## 3. 系统定位

Verge Browser 需要同时满足两条使用路径：

- 人类路径：通过 noVNC/VNC 进入浏览器，处理扫码、登录、验证码等必须人工完成的步骤。
- Agent 路径：通过 Playwright over CDP、截图、GUI action、文件接口完成自动化操作。

这两条路径服务于同一个 sandbox，而不是两个独立产品。

## 4. 当前实现基础

仓库已经具备可运行的底层能力：

- Runtime 容器中运行 Chromium、Xvfb、Openbox、x11vnc、websockify。
- API Server 已支持 sandbox 的创建、删除、暂停、恢复、浏览器重启。
- 已支持 CDP 代理、VNC ticket、截图、GUI action、文件读写与下载上传。
- 已有 runtime 相关单元测试与集成测试。

因此，当前主要短板不是底层运行时，而是上层产品入口与 agent 接入层。

## 5. 核心问题

当前实现与目标之间的关键差距有五项：

1. 鉴权模型不符合要求：业务 API 不能允许匿名访问。
2. 缺少管理入口：没有管理页，也缺少面向 dashboard 的列表接口。
3. 缺少实例标识抽象：缺乏正式的 `alias` 字段和统一实例管理视图。
4. 缺少 agent 入口层：没有 SDK、CLI、稳定 JSON 输出和标准工作流封装。
5. 人机协同还未产品化：技术能力存在，但没有被组织成简单可复用的操作路径。

## 6. 关键技术决策

### 6.1 鉴权

采用单一 admin token 模型。

- 服务端通过配置项注入固定 admin token。
- 所有业务 API 默认要求 `Authorization: Bearer <token>`。
- token 缺失或错误时返回 `401`。
- `healthz` 一类探活接口可以匿名访问。
- 不引入 JWT 用户体系、刷新 token、RBAC。

CLI 和 SDK 默认从 `VERGE_BROWSER_TOKEN` 读取 token，也允许命令行或构造参数覆盖。

### 6.2 资源模型

继续以 sandbox 为主资源，保持现有 URL 模型：

- `GET /sandboxes`
- `POST /sandboxes`
- `GET /sandboxes/{sandbox_id}`
- `PATCH /sandboxes/{sandbox_id}`
- `DELETE /sandboxes/{sandbox_id}`
- 其余 browser、vnc、files 等子路由继续挂载在 `/sandboxes/{sandbox_id}/...`

`alias` 是对 sandbox 的可读入口，不替代内部主键。

建议 sandbox 记录最少包含：

- `id`
- `alias`
- `status`
- `created_at`
- `updated_at`
- `last_active_at`
- `metadata`

### 6.3 管理页

管理页先做 MVP，不做复杂后台。

首版只需要：

- token 登录页
- sandbox 列表页
- sandbox 详情页
- 基础操作：创建、暂停、恢复、删除、打开 VNC、查看 CDP 信息

建议前端技术栈为 `Vite + React`。

### 6.4 SDK 与 CLI

SDK/CLI 的目标不是简单包一层 REST，而是缩短 agent 工作流。

必须封装的内容：

- token 注入
- alias 到 sandbox_id 的解析
- VNC ticket 申请
- CDP URL 获取
- 常见错误映射
- 适合 agent 消费的稳定 JSON 输出

CLI 名称固定为 `verge-browser`。

### 6.5 高层 agent 能力

参考 `agent-browser` 的方向，但分阶段建设。

第一阶段只做基础可用能力：

- 创建和查找 sandbox
- 获取 CDP/VNC/截图/文件能力
- 统一 JSON 输出

第二阶段再考虑高级 agent-first 能力：

- snapshot
- semantic locator
- 引用式元素操作
- 更高层的浏览器动作抽象

## 7. 统一架构

下一阶段系统由四层组成：

### 7.1 Runtime 层

职责：

- 启动 Chromium/Xvfb/Openbox
- 暴露 CDP、noVNC、共享工作区
- 支撑 pause/resume/restart 等生命周期动作

这一层沿用现有实现，不做架构重写。

### 7.2 Control Plane API 层

基于 FastAPI，职责：

- admin token 校验
- sandbox 生命周期管理
- browser/files/vnc 等控制面 API
- 为管理页和 SDK/CLI 提供稳定 JSON 接口

### 7.3 Admin Web UI 层

新增 Web 管理界面，职责：

- token 登录和保存
- sandbox 列表与详情展示
- VNC 入口跳转
- 展示状态、CDP 信息和基础元数据

### 7.4 Agent Access 层

新增 SDK 与 CLI，职责：

- 提供适合脚本、agent、skills 的统一入口
- 封装 VNC/CDP/生命周期等低层细节
- 后续承接 snapshot 和语义操作能力

## 8. 典型工作流

### 8.1 人机协同工作流

1. Agent 通过 SDK/CLI 创建 sandbox。
2. Agent 打开页面并执行初始自动化。
3. 如遇验证码、扫码、账号登录，Agent 申请 VNC ticket 并生成可交给人类的接管链接。
4. 人类通过 noVNC 完成操作。
5. Agent 再通过 CDP 或 API 继续后续流程。

### 8.2 管理台工作流

1. 部署者配置 admin token 并启动服务。
2. 管理员在 Web UI 输入 token 登录。
3. 查看全部 sandbox 状态与基础信息。
4. 对目标 sandbox 执行创建、查看、暂停、恢复、删除、打开 VNC 等操作。

## 9. API 演进要求

新增接口时遵循以下规则：

- 不破坏现有 `/sandboxes/{sandbox_id}/...` 主轴。
- 管理页和 CLI 优先复用同一组 REST 接口。
- 面向 agent 的输出应提供稳定字段和机器可读错误。
- alias 相关查询可以在 API 层支持，但内部仍以 `sandbox_id` 为准。

建议补充的 API 能力：

- `GET /sandboxes` - sandbox 列表
- `GET /sandboxes/{sandbox_id}` - sandbox 详情聚合信息
- `POST /sandboxes` - 创建（支持 alias）
- `PATCH /sandboxes/{sandbox_id}` - alias/metadata 更新
- 基础统计信息

## 10. 实施优先级

### P0

- 强制 admin token 鉴权
- sandbox 列表与详情接口
- alias 字段与更新能力
- Web 管理页 MVP

### P1

- Python SDK
- `verge-browser` CLI
- Playwright/CDP 集成示例
- 人机协同工作流封装

### P2

- snapshot
- semantic locator
- 引用式浏览器操作
- 面向 skills 的高层命令集

## 11. 验证原则

涉及 runtime 或控制面改动时，验证应同时覆盖 API Server 与 Runtime。

预期验证命令：

```bash
docker build -f docker/runtime-image.Dockerfile -t verge-browser-runtime:latest .
. .venv/bin/activate
PYTHONPATH=apps/api-server pytest tests/unit tests/integration/test_runtime_api.py
```

如果本地无法运行 Docker，至少应保证单元测试通过。
