# T2: 面向项目愿景的现状梳理、待办与技术方案

## 1. 文档目的

`docs/spec/req.md` 给出的愿景很明确，这个项目最终不是单纯的 browser sandbox API，而是一个可自部署、有人类接管入口、也对 agent 足够友好的浏览器基础设施。

当前仓库已经完成了一部分底层能力，但产品形态仍停留在“控制面 MVP”。本文档的目标是：

1. 以 `req.md` 作为优先级来源，重新定义接下来的建设重心。
2. 基于当前代码真实实现情况，梳理“已经有了什么 / 还缺什么”。
3. 给出一份能直接指导后续开发的待办清单与技术落地建议。

本文档补充参考了 [`vercel-labs/agent-browser`](https://github.com/vercel-labs/agent-browser) 的产品思路，并将其视为 agent 侧体验的重要对标：

- `agent-browser` 更偏 agent-first 的本地浏览器 CLI
- Verge Browser 更偏 self-hosted 的远程 browser sandbox 平台

因此，这里不是只借鉴“方法论”，而是要求 Verge Browser 在 agent 使用侧尽量具备与 `agent-browser` 同等级的核心能力与工作流体验；差异主要体现在底层运行形态是远程 sandbox，而不是本地 daemon。

本文档默认和以下文件一起阅读：

- `docs/spec/req.md`
- `docs/tech.md`
- `docs/api.md`
- `apps/api-server/app/routes/`
- `apps/api-server/app/services/`
- `apps/sandbox-runtime/scripts/`

## 2. 愿景重述

从 `docs/spec/req.md` 可以提炼出 4 条主目标：

### 2.1 可自部署

用户应能自己部署整套服务，并通过简单配置启用管理能力和鉴权能力。核心诉求不是复杂 IAM，而是“部署后立即可用”的 admin token 模式。

### 2.2 可被人管理

需要一个管理页面。用户提供 admin auth token 后，可以看到自己所有实例的状态，并进行基础操作。

这里的“实例”在当前代码语境中，实际对应 sandbox。

### 2.3 可被 agent 方便使用

鉴权、连接、创建 sandbox、获取 VNC/CDP/文件能力，这些动作不能要求 agent 逐个拼 REST 请求。需要：

- SDK
- 面向 agent 的 CLI
- 后续可包装成 skills 的稳定接口

这里还应再加一条更强的要求：

- 对 agent 的实际使用体验，应尽量对齐 `agent-browser`

也就是说，agent 侧不应该只拿到“创建 sandbox 的 API”，还应拿到足够高层的浏览器操作抽象，例如：

- 稳定的会话概念
- 面向模型的结构化页面观察结果
- 可复用的动作引用或元素定位能力
- 适合脚本和 skills 的 CLI 输出

### 2.4 支持人机协同

人类用户的主要使用方式是：VNC + agent 协作。

也就是说，产品最终需要同时优化两条链路：

- Human loop：用户打开 noVNC，手动处理扫码、账号认证、验证码等环节
- Agent loop：agent 通过 Playwright/CDP/文件接口进行自动化操作

这不是两个分离产品，而是同一个 sandbox 的两种控制面。

## 3.2 还没有落地的关键能力

这些缺口不是“锦上添花”，而是当前实现和愿景之间最关键的断层。

### A. 没有收口到极简固定 token 鉴权

当前 API 的认证行为是：

- `Authorization` 头是可选的
- 不带 token 时 subject 默认是 `anonymous`
- 只做 JWT 解码，没有固定 admin token 配置
- 未形成“未带 token 直接 401”的统一行为

这意味着当前鉴权既不够严格，也不够简单，不符合“固定 token、默认拒绝未授权请求”的目标。

对应代码：

- `apps/api-server/app/deps.py`
- `apps/api-server/app/auth/jwt.py`
- `apps/api-server/app/config.py`

### B. 没有管理页面

仓库中没有 Vite + React 前端工程，也没有实例管理 UI。

当前 noVNC 页面只是 sandbox 内部接管页面，不是控制台。

### C. 没有 sandbox 列表 / 管理 API

虽然 registry 支持 `all()`，但对外没有：

- 列表接口
- 筛选接口
- 聚合状态接口
- 管理页使用的 dashboard API

现在 API 更像“你已经知道 sandbox_id 才能操作”的控制平面。

### D. 没有 agent-friendly SDK / CLI

仓库中还没有：

- Python SDK
- TypeScript SDK
- CLI
- Playwright 辅助封装
- skill 集成入口

这会直接导致 agent 使用门槛过高。

更准确地说，当前缺的不只是“SDK/CLI 这个包装层”，还缺一整层 agent-first 交互能力。对照 `agent-browser` 的使用体验，当前还没有：

- 命名会话或持久 session 的产品化抽象
- 结构化页面快照能力
- 面向模型的元素引用机制
- 面向 agent 的高层操作命令
- 统一的 `--json` 机器输出协议

### E. 人机协同流程还没被产品化

当前已经有技术能力，但没有形成稳定工作流抽象，例如：

- agent 创建 sandbox
- agent 请求 VNC ticket 并把接管链接交给人类
- 人类完成认证
- agent 再通过 Playwright/CDP 继续执行

这条主流程需要 SDK/CLI 和文档一起封装出来，否则只是“功能存在”，还不是“好用”。

### F. 多实例管理能力不足

从愿景看，用户提供 token 后应当能看到“自己所有实例情况”。而当前系统还缺少：

- 面向管理页的列表视图数据
- sandbox 标签/用途/owner 字段约束
- 基础统计信息
- 长时间运行实例的清理策略

### G. 部署体验仍偏工程向

当前可以本地运行，也有 Dockerfile/compose，但要满足“用户自己部署且只配置 admin auth token”，还需要进一步补齐：

- 更明确的生产部署文档
- 首次启动配置项说明
- 反向代理和 HTTPS 建议
- 前后端一体部署方式

## 4. 愿景对照表

| 愿景目标 | 当前状态 | 结论 |
|---|---|---|
| 自部署 + 简单鉴权 | 有基础配置系统，但没有固定 token + 强制 401 模型 | 未完成 |
| 管理页面 | 不存在 | 未开始 |
| 管理所有实例 | 只有按 ID 操作 API，没有列表/面板 API | 未完成 |
| 给 agent 方便使用 | 有底层 REST/WS 能力，但没有 SDK/CLI | 未完成 |
| 人类通过 VNC 接管 | 技术能力已具备 | 已完成 MVP |
| agent 通过 Playwright 自动化 | CDP 代理已具备，能被 Playwright 连接 | 已完成 MVP |
| 人机协作闭环 | 缺少统一流程封装与上层产品入口 | 部分完成 |

判断上可以很明确：当前项目的强项是 runtime 和底层控制接口，弱项是“产品入口层”和“agent 接入层”。

如果进一步按 `agent-browser` 作为对标来看，当前差距主要不是“能不能控制浏览器”，而是“能不能让 agent 以很短的上下文成本稳定控制浏览器”。

## 5. 下一阶段建设原则

为了贴近 `req.md`，接下来不建议继续只做底层能力横向扩展，而应把优先级切到以下顺序：

1. 先补“可用入口”，即极简 admin token + 管理 API + 管理页面。
2. 再补“可用 agent 入口”，即 SDK + CLI + Playwright 接入封装。
3. 最后再扩展高级能力，如更复杂的权限模型、MCP、云原生调度。

简化说：

- 现在最缺的不是再多一个 browser endpoint
- 而是把已有能力组织成“人能用、agent 也能用”的产品

这点和 `agent-browser` 的经验是一致的：

- 对 agent 来说，真正重要的不是 API 数量
- 而是主工作流是否足够短、稳定、可脚本化

因此 Verge Browser 接下来应优先建设 agent-first 交互层，并让 agent 侧能力尽量覆盖 `agent-browser` 已验证有效的那一套核心体验；不是继续增加零散底层接口，也不是只停留在“提供远程浏览器 API”。

## 6. 待办清单

## 6.1 P0: 必须优先完成

### 1. Admin token 鉴权模型（极简实现）

目标：

- 部署时配置单一 `VERGE_ADMIN_AUTH_TOKEN`
- 所有请求强制校验，不再允许匿名
- CLI/SDK 优先从环境变量 `VERGE_BROWSER_TOKEN` 读取

具体实现：

- 服务端必填 `VERGE_ADMIN_AUTH_TOKEN`，无默认值
- 请求头 `Authorization: Bearer <token>`，错误或未带均返回 401
- 仅 `/healthz` 允许匿名访问
- CLI/SDK 读取 `VERGE_BROWSER_TOKEN` 环境变量作为默认 token
- 命令行支持 `--token` 覆盖环境变量

明确不做：

- JWT、账号体系、多角色、token 刷新等复杂能力

### 2. Sandbox 列表与详情增强 API

至少需要新增：

- `GET /sandboxes`
- `GET /sandboxes/{sandbox_id}`
- `POST /sandboxes`
- `DELETE /sandboxes/{sandbox_id}`
- `POST /sandboxes/{sandbox_id}/pause`
- `POST /sandboxes/{sandbox_id}/resume`
- `POST /sandboxes/{sandbox_id}/browser/restart`

其中新增重点是列表接口返回：

- id
- alias
- status
- created_at / updated_at
- viewport
- default_url
- metadata
- runtime error
- 是否可 VNC 接管
- 是否可 CDP 连接

当前 `SandboxRecord` 里并没有明确保存 `alias`、`default_url`、owner、last activity 等字段，建议补充模型。

### 3. 管理页面 MVP

技术选型直接采用 `req.md` 提到的：

- Vite
- React

MVP 页面建议只做三块：

1. token 输入 / 保存页
2. Sandbox 列表页
3. Sandbox 详情页

详情页至少包含：

- 状态信息
- 创建时间
- VNC 打开按钮
- CDP 地址展示
- 文件工作区入口说明
- 操作按钮：pause/resume/restart/delete

### 4. 统一接管入口

管理页需要把当前的 VNC ticket 流程包装成对人类可用的操作：

- 点击“Open VNC”
- 前端请求 `/vnc/tickets`
- 自动打开 `/vnc/?ticket=...`

这样人类用户不需要理解 ticket 模式。

### 5. 生产部署文档 MVP

需要一份明确文档说明：

- 如何配置 admin token
- 如何启动 API server
- 如何构建 runtime image
- 如何部署前端
- 如何通过反向代理暴露服务

没有这部分，项目很难达到“用户可以自己部署”的目标。

## 6.2 P1: 紧随其后

### 6. Agent SDK

优先建议先做 Python SDK，因为当前控制面本身是 Python，和 agent 生态也更贴近。

建议 SDK 提供高层对象：

- `VergeClient`
- `SandboxHandle`

核心方法例如：

- `create_sandbox()`
- `list_sandboxes()`
- `get_sandbox(id_or_alias)`
- `pause() / resume() / delete()`
- `create_vnc_url()`
- `cdp_ws_url()`
- `read_file() / write_file() / upload_file()`
- `screenshot()`
- `actions()`

目标不是 1:1 暴露 REST，而是把 ticket、URL 拼接、别名解析、错误处理、重试等细节包掉。

### 7. Agent-first 浏览器操作层

如果以 `agent-browser` 为对标，单纯提供“会话管理 CLI”还不够，还需要补一层真正给 agent 用的浏览器操作层。

建议能力包括：

- 页面结构化快照
- 面向模型稳定的元素引用
- 基于引用的点击、输入、选择、滚动
- 文本提取、链接提取、标题提取
- tab / page 管理
- 会话级持久 profile 复用

这一层不一定直接暴露为 FastAPI 原子接口，也可以建立在远程 Playwright/CDP 连接之上，由 SDK/CLI 负责封装。

### 8. Agent CLI

CLI 是给 skills 和自动化脚本用的最短路径。

这里建议明确参考 `agent-browser` 的几个实践：

- CLI 优先，而不是先做很厚的 SDK
- 所有核心命令尽量支持 `--json`
- 输出形状稳定，避免自然语言噪声
- 命令语义贴近 agent 的行动单元

建议命令结构：

```bash
# token 设置（两种方式，按优先级）
export VERGE_BROWSER_TOKEN=<token>   # 推荐：环境变量
verge-browser --token <token> sandbox ls     # 命令行参数覆盖环境变量

# sandbox 管理
verge-browser sandbox create --alias shopping
verge-browser sandbox ls
verge-browser sandbox get <id-or-alias>
verge-browser sandbox pause <id-or-alias>
verge-browser sandbox resume <id-or-alias>
verge-browser sandbox rm <id-or-alias>

# VNC/CDP
verge-browser sandbox vnc <id-or-alias>      # 自动申请 ticket 并打开浏览器
verge-browser sandbox cdp <id-or-alias>      # 输出 CDP WebSocket URL

# 浏览器操作（agent-first）
verge-browser page snapshot <id-or-alias> --json
verge-browser element click <id-or-alias> --ref ...
verge-browser element type <id-or-alias> --ref ... --text ...
verge-browser page extract <id-or-alias>
```

CLI 设计原则：

- 非交互式优先，适合脚本调用
- 所有关键命令支持 `--json` 输出
- 稳定的退出码（0=成功，非0=失败）
- 错误信息输出到 stderr，不污染 stdout JSON

相比 `agent-browser`，Verge Browser 的 CLI 不只是要有 DOM / page / element 这一层，还要额外解决”远程 sandbox 会话管理 + 获取接入入口”这一层：

- 创建和列出 sandbox
- 通过 alias 或 id 获取 sandbox
- 获取 CDP 连接信息
- 生成或直接打开 VNC 接管链接
- 管理 profile/workspace 生命周期
- 输出给 Playwright 或 skills 直接消费的 JSON

CLI 分层设计：

- **sandbox 层**：负责远程会话生命周期
- **page/element 层**：负责 agent-first 的浏览器交互

### 9. Playwright 接入文档与辅助封装

愿景明确提到 agent + Playwright，因此至少需要：

- 用 SDK/CLI 获取 CDP WebSocket 地址
- 提供官方示例，展示如何 `connect_over_cdp`
- 说明何时用 CDP，何时让人类接管 VNC

此外建议明确一条实现策略：

- 底层仍以 Playwright/CDP 作为主要浏览器自动化内核
- 但对 agent 暴露的接口，应优先是更稳定、更短上下文的高层命令

### 10. Sandbox 元数据增强

建议补充字段：

- `alias`
- `name`
- `owner`
- `default_url`
- `tags`
- `last_active_at`
- `last_runtime_error`

其中 `alias` 应作为用户区分和获取 sandbox/browser 信息的主要人类可读标识，CLI 和 SDK 都应支持 `id_or_alias` 输入。

这样管理页、SDK、清理策略才有足够信息。

## 7. 推荐技术方案

## 7.0 从 `agent-browser` 借鉴什么

结合 `vercel-labs/agent-browser` 的公开说明，可以提炼出几条对 Verge Browser 很重要的经验：

### 1. CLI 是 agent 产品面的核心，不是附属品

`agent-browser` 的主入口就是 CLI，并且安装、连接、操作、输出格式都围绕 CLI 设计。  
对 Verge Browser 来说，这意味着 CLI 不应只是调试工具，而应成为：

- SDK 之上的标准操作层
- skills 的稳定承载层
- 文档示例的第一入口

### 2. 机器可消费输出要作为一等能力

`agent-browser` 明确强调 `--json` 输出和供模型复用的稳定引用工作流。  
Verge Browser 应尽量对齐这种模式，而不只是做“会话管理 JSON API”。因此至少应让以下命令具备稳定 JSON 输出：

- `sandbox create`
- `sandbox ls`
- `sandbox get`
- `page snapshot`
- `page extract`
- `element click`
- `element type`
- `sandbox cdp`
- `sandbox open-vnc`
- `sandbox files *`

同时，这些命令在输入参数上应同时支持 `sandbox_id` 和 `alias`。

### 3. 会话持久化要成为正式概念

`agent-browser` 明确区分 session、profile、session-name，并强调登录态复用。  
Verge Browser 当前实际上已经通过 `/workspace/browser-profile` 具备这个基础，但还没有被产品化表达出来。

因此文档、SDK、CLI 中都应显式表达：

- sandbox 就是远程持久会话
- `browser-profile` 是登录态载体
- pause/resume 是会话生命周期管理，而不只是容器操作

### 4. 人机协同应作为正式工作流而不是附带能力

`agent-browser` 提到 live preview / pair browsing。  
Verge Browser 当前的 noVNC 更适合承载这类场景，因此应把以下流程写成标准路径：

1. agent 创建 sandbox
2. agent 将 VNC 入口交给人类
3. 人类完成验证码、扫码或认证
4. agent 继续通过 CDP / Playwright 自动化

### 5. 需要提供面向模型的页面观察与操作抽象

如果 agent 侧要真正对齐 `agent-browser`，则不能只停留在 screenshot + xdotool。还需要补上：

- 页面结构化观察结果
- 稳定元素引用
- 基于引用的高层操作

这些能力可以由远程 Playwright 会话生成，但在产品面上应作为 Verge Browser 自己的 agent interface 提供。

### 6. skills / AGENTS.md 集成应尽早考虑

`agent-browser` 不只提供工具，还提供给 Claude Code 等 agent 的接入说明。  
这对 Verge Browser 很关键，因为 `req.md` 已明确提到“可以进一步包装为 skills”。

因此 Verge Browser 的 CLI 从一开始就应满足：

- 非交互
- 稳定 JSON
- 可通过单条命令完成核心动作
- 容易写进 `AGENTS.md` 或 skill 指令中

### 7. 不建议直接照搬的部分

以下能力可以参考，但需要按远程 sandbox 形态重做，而不是直接迁移实现：

- 本地守护进程式单机会话管理
- Rust native CLI 路线

原因是 Verge Browser 的核心优势不在“本地浏览器指令集”，而在“远程 self-hosted sandbox + VNC/CDP 双入口”。

所以正确方向不是弱化 `agent-browser` 那套 agent 交互层，而是把它迁移到远程 sandbox 架构中，做成：

- 更适合远程部署
- 更适合人机协同接管
- 更适合把 sandbox 当作持久浏览器资产管理

## 7.1 鉴权方案（极简设计）

### 核心原则

鉴权只做最小实现，不追求产品级 IAM。目标是"简单部署即可用"，而非"支持复杂权限模型"。

### 方案：单一 Admin Token

**服务端配置**:
- 环境变量 `VERGE_ADMIN_AUTH_TOKEN` 设置唯一 token
- 未设置该变量时，服务启动失败（明确报错）

**请求方式**:
```
Authorization: Bearer <admin-token>
```

**校验规则**:
- Token 错误：401 Unauthorized
- 未携带 Token：401 Unauthorized
- **不再默认 anonymous，不再允许匿名访问**

**例外**:
- `/healthz` 允许匿名（用于部署健康检查）
- 其他所有控制面接口强制要求 token

### CLI/SKD 使用方式

**环境变量优先**（推荐）:
```bash
export VERGE_BROWSER_TOKEN=<admin-token>

verge-browser sandbox ls                # 自动读取环境变量
verge-browser sandbox create --alias a  # 无需每次 --token
```

**命令行参数覆盖**:
```bash
verge-browser --token <token> sandbox ls
```

### 为什么这样设计

1. **极小实现成本**: 服务端只需一个 string 比对，无需 JWT、无需数据库
2. **明确的部署契约**: 部署时必须配 token，不配则启动失败
3. **单一用户场景足够**: 自部署用户通常就是 admin，不需要多角色
4. **CLI 体验好**: 配置环境变量后，所有命令免参数使用

### 服务端改造（极简）

```python
# app/config.py
admin_auth_token: str  # VERGE_ADMIN_AUTH_TOKEN，无默认值，必填

# app/auth/simple.py
def require_admin_token(authorization: str = Header(...)) -> None:
    """极简校验：直接字符串比对"""
    scheme, _, token = authorization.partition(" ")
    if scheme.lower() != "bearer" or not token:
        raise HTTPException(401, "invalid authorization")
    if token != get_settings().admin_auth_token:
        raise HTTPException(401, "invalid token")
```

### 明确不做的事

鉴权保持极简，以下能力**明确不在当前阶段**:
- ❌ JWT 登录/登出流程
- ❌ 用户注册/账号体系
- ❌ 多 token/多角色
- ❌ 权限细分（read-only vs admin）
- ❌ token 过期/刷新机制
- ❌ audit log

这些能力未来需要时，可以平滑升级，但当前要保持最小可行。

## 7.5 API 演进建议

建议在保持当前 `/sandboxes/{sandbox_id}/...` 路由模型不变的前提下补齐：

### 新增接口

- `GET /sandboxes`

`GET /sandboxes` 返回管理页和 CLI 最需要的聚合信息。

此外建议在现有 `/{sandbox_id}` 风格不变的前提下，服务端依赖层支持将路径参数解析为 `id_or_alias`，从而允许用户和 CLI 用 alias 获取 browser 信息而不必记住真实 id。

### 保持不变的接口

- `/sandboxes/{sandbox_id}/browser/*`
- `/sandboxes/{sandbox_id}/vnc/*`
- `/sandboxes/{sandbox_id}/files/*`

原因：

这些路径已经和 `docs/tech.md` 的设计一致，不应为管理页面重构掉底层 API 结构。

## 9. 关键判断

接下来最重要的产品判断只有一条：

**不要再把重点放在“补更多底层 browser endpoint”，而要把重点放在“把现有能力组织成可部署、可管理、可供 agent 调用的产品入口”。**

换句话说，当前项目并不缺“浏览器控制原子能力”，真正缺的是：

- 一个简单可信的 admin auth token 模式
- 一个管理所有 sandbox 的控制台
- 一个把 REST/WS 细节封装掉的 SDK/CLI

只要这三件事补上，当前代码基础已经足以支撑 `req.md` 描述的主路径。

## 10. 结论

当前项目已经完成了 browser sandbox 的底座，尤其是：

- runtime 组织方式
- VNC / CDP / files / GUI actions
- lifecycle 与持久化恢复

这些都说明技术方向是对的。

但如果以 `docs/spec/req.md` 为最终目标，下一阶段的核心不应该再是“继续打磨底层沙箱”，而应该是把这套能力产品化为三层入口：

1. 面向部署者的 admin token
2. 面向人类操作者的 Web 控制台
3. 面向 agent 的 SDK / CLI

这也是本文档给出的待办优先级。
